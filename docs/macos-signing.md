# macOS code-signing + Sparkle auto-update

This is the cookbook for cutting a signed, notarized, auto-updating
PixCull release on macOS. Until this is set up, brew users have to
manually `brew upgrade pixcull` every time; with Sparkle wired in,
the app pings the appcast on launch and prompts the user to install
new versions.

## 0. Prerequisites (one-time)

- **Apple Developer Program membership** — $99/year. PixCull
  uses the Developer ID program (NOT the Mac App Store), which
  lets us distribute outside the App Store but still get
  notarization.
- **Apple Developer ID Application certificate** — generated from
  developer.apple.com under "Certificates, Identifiers & Profiles".
  Install in your Keychain; `security find-identity -v -p
  codesigning` should list it.
- **App-specific password for `notarytool`** — appleid.apple.com →
  Sign-In and Security → App-Specific Passwords. Store in
  `~/.config/pixcull/notarytool-cred` (chmod 600) along with your
  team ID and Apple ID email.
- **Sparkle binaries** — clone <https://github.com/sparkle-project/Sparkle>
  and extract `bin/generate_keys` + `bin/sign_update` somewhere
  on `$PATH`. The EdDSA key pair generated by `generate_keys`
  becomes the *update signing key* — distinct from the macOS
  code-signing identity; protects against MITM attacks on the
  download CDN.

## 1. EdDSA key pair (one-time per app)

```sh
generate_keys                 # writes the private key to your Keychain
generate_keys --account pixcull -p  # prints the matching public key
```

Copy the public key into `Info.plist`:

```xml
<key>SUPublicEDKey</key>
<string>BASE64_PUBLIC_KEY_HERE</string>
```

…and into the `--ed-key-file` of any CI step that signs releases.
**Keep the private key in your Keychain ONLY.** Anyone with the
private key can push malicious "updates" to every PixCull user.

## 2. Per-release pipeline

```sh
# 1. Build the .app (PyInstaller / py2app — out of scope here).
make app                                  # → dist/PixCull.app

# 2. Code-sign with hardened runtime (required for notarization).
codesign --deep --force --options runtime \
    --entitlements dist/entitlements.plist \
    --sign "Developer ID Application: Your Name (TEAMID)" \
    dist/PixCull.app

# 3. Package as DMG (use create-dmg or hdiutil).
create-dmg --volname "PixCull 0.7" \
    --background dist/dmg-bg.png \
    --window-size 600 400 \
    --icon-size 128 \
    --app-drop-link 425 200 \
    dist/PixCull-0.7.0.dmg dist/PixCull.app

# 4. Notarize.
xcrun notarytool submit dist/PixCull-0.7.0.dmg \
    --keychain-profile pixcull-notary \
    --wait
xcrun stapler staple dist/PixCull-0.7.0.dmg

# 5. EdDSA-sign the DMG (this signature goes IN the appcast).
sign_update dist/PixCull-0.7.0.dmg \
    --account pixcull \
    > dist/PixCull-0.7.0.sig
# sign_update prints something like:
#   sparkle:edSignature="MEUCIQD..." length="48319201"
# Capture both into releases.json.

# 6. Update dist/releases.json with the new entry — see
#    dist/releases.example.json for the schema.

# 7. Regenerate the appcast.
python scripts/build_appcast.py dist/releases.json --out dist/appcast.xml

# 8. Upload BOTH the DMG and the appcast to the CDN.
aws s3 cp dist/PixCull-0.7.0.dmg s3://pixcull-dist/downloads/
aws s3 cp dist/appcast.xml       s3://pixcull-dist/appcast.xml \
    --cache-control "max-age=300, public"
```

The 5-minute cache on `appcast.xml` is intentional: Sparkle polls
the appcast on launch + every 24h by default, so a 5-minute stale
window is invisible to the user but stops CloudFront from holding
a bad release file for an hour after a hot-fix.

## 3. Wiring Sparkle into the .app

The `.app` bundle's `Info.plist` needs three Sparkle keys:

```xml
<key>SUFeedURL</key>
<string>https://pixcull.app/appcast.xml</string>
<key>SUEnableInstallerLauncherService</key>
<true/>
<key>SUPublicEDKey</key>
<string>BASE64_PUBLIC_KEY_FROM_STEP_1</string>
```

…and `Sparkle.framework` must be embedded in `Contents/Frameworks/`.
See <https://sparkle-project.org/documentation/> for the framework
embed details.

## 4. Releasing a critical update

For a data-loss or security fix, set `"critical": true` in the
releases.json entry. The appcast emits `<sparkle:criticalUpdate />`
which makes Sparkle disable "Skip this version" / "Remind me later"
— the user must install or quit the update prompt.

Use sparingly. False-alarm critical updates train users to dismiss
them on autopilot, which defeats the point.

## 5. Brew tap (post-Sparkle)

Once Sparkle is wired in, the brew formula still ships first-time
installs (homebrew/cask). But updates beyond that come from Sparkle
directly — much faster for users (no `brew upgrade` round-trip)
and gives us control over rollout (we could phase a release to
1% → 10% → 100% by sharding the appcast in front of CloudFront,
though that's v0.8 work).

The cask formula keeps a stable URL pattern so a fresh install
always gets the latest:

```ruby
url "https://pixcull.app/downloads/PixCull-#{version}.dmg"
```

## 6. Troubleshooting

- **"App is damaged and can't be opened"** — the app wasn't
  notarized, or stapler hasn't been re-run after notarytool
  finished. Re-staple and re-distribute.
- **Sparkle says "Update file is missing / unreadable"** — the
  `sparkle:edSignature` in the appcast doesn't match what
  `sign_update` printed at release time. Re-sign + re-publish
  the appcast.
- **No update prompt** — Sparkle only polls every 24h by default.
  Force a check: `defaults write app.pixcull.PixCull SUEnableAutomaticChecks
  -bool YES && defaults write app.pixcull.PixCull SULastCheckTime
  -date "yesterday"` then relaunch.
