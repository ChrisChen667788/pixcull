# v2.24 — 性能:图片内存虚拟化(离屏缩略图停靠)

> 迭代方案里排的性能债"真·滚动虚拟行 (10k+)"的关键一刀。承接 v2.18 渐进水合
> 与 P-UX-18 首屏占位——本轮补上**已解码图片内存**的边界。

## 问题

P-UX-18(≥500 行的大批量)用 IntersectionObserver 把**首屏卡片 DOM** 限住:
只急渲染前 100 张,其余发占位符,滚到附近再物化。但**物化后卡片和它的已解码
缩略图永久驻留**——把一场 10k 婚礼滚到底,就把 10k 张 JPEG 解码进了内存。
`loading="lazy"` 只延迟首次加载,从不回收。

## 方案(v2.24-P0)

给大批量路径加**第二个 observer**,把已解码缩略图限制在视口周围的窗口内:

- 卡片远离视口 **>3 屏**(rootMargin `300% 0px`)→ **停靠**其 `.thumb` 图片:
  `src` 移到 `data-parked-src`、清空 `src`,浏览器回收解码。
- 卡片**回到附近** → 从 `data-parked-src` 恢复 `src`,重新加载。
- **卡片元素本身、决策徽章、键盘索引、焦点全不动**——只切 `<img src>`;
  `.thumb-wrap` 的 `aspect-ratio: 4/3` 撑住高度,滚动条不跳。
- 物化回调里给新物化的卡片挂上停靠 observer;首批急渲染的卡也挂上。
- 关键:**避开脆弱的决策/渲染路径**(v2.13"一处破九处炸"的教训)——不做
  卡片去物化(那会牵动决策状态/键盘/焦点),只回收图片解码内存,拿下大头。

lightbox 不受影响:`openLightbox` 从 rows 查行、自建全图 URL,不读 thumb src。

## 验收(实测)

600 行合成 run,Playwright 滚动测**已加载 vs 已停靠**缩略图:
- 顶部:100 卡物化,live src **48** / parked 52
- 滚动后:物化增至 **352** 卡,live src **恒定 76**、parked 276
- 回顶部:live src 回落 **48**(顶部图恢复),parked 304

**已加载图数量始终 ~48-76(视口窗口),与滚动距离/物化卡数无关**。没这刀
live src 会等于总卡数一路涨到 600。图片内存被限住,与 run 大小解耦。

- 门禁绿(5 预期 skip);test_hydration 加图片停靠钩子守卫;版本 2.23→2.24 lockstep。

## 后续候选(纯代码功能/性能)

  只差 `pinForCompare` 选到 2 张就立即开、无法累积 3-5 张;需新增选择托盘 +
  键位,改动自由配对的肌肉记忆,值得单独一轮验证。

  节点数——但牵动决策/键盘/焦点,高风险,需按"从当前 row 状态重建"做对。

  拆分(card/filters 模块)——纯架构,低风险。
