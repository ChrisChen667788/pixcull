# v2.26 — 性能进阶:真去物化(远处卡片 DOM 回收成占位符)

> 迭代队列 #2(性能进阶)。承接 v2.24 图片内存虚拟化——那一刀限住了已解码
> 图片内存;这一刀再限住**卡片 DOM 节点数**,做到真正的窗口化虚拟滚动。

## 问题

v2.24 后:大 run 滚动时已解码图片被限在视口窗口,但**物化的卡片 DOM 节点**
永久驻留——滚一遍 10k 婚礼就攒出 ~10k 张卡的 DOM(每张 ~30 元素 = 30 万节点),
样式/布局重算变重。P-UX-18 只物化不回收。

## 方案(v2.26-P0)

在既有 P-UX-18 物化观察者旁加**去物化**:

- 卡片远离视口 **>5 屏**(rootMargin `500%`,远宽于 ~200% 物化边距 → 300% 迟滞带
  防边界抖动)→ **拆回占位符**(测得高度、带 data-idx),`_io.observe` 重新盯它。
- 卡片回到附近 → **从当前 row 重物化**:`renderCard(segRows[idx])` 而非冻结的
  `segments[idx]` 字符串——**这是正确性关键**:去物化期间(或之前)做的判定改动
  写在 `r.decision` 上,segRows[idx] 就是 rows[] 里那个对象,重物化即反映当前判定,
  不会回退。首批急渲染的 100 张标 segIdx=-1 永不回收(固定小成本)。
- 三观察者协作:物化(~200%)→ 图片停靠(300%,v2.24)→ 去物化(500%);
  重渲染时全部 disconnect。键盘导航本就只遍历物化卡(`visibleCards()`),回收远处
  卡不改变视口附近的可达集。

## 验收(Playwright 实测,600 行 run)

- 顶部:100 卡 / 500 占位符。
- **滚遍全部 600 行 → 仍只 172 卡**(不是 600!)—— DOM 节点数被限住。
- **回到顶部 → 恢复 100 卡 / 500 占位符** —— 双向回收都工作,顺序/文件名正确,
  零 JS 错误。
- 判定持久:culling 一张卡 → 类为 cull → 复检仍 cull(r.decision 更新持久)。
  segIdx≥100 卡的重物化走**完全相同**的 `renderCard(segRows[idx])` 路径,无代码
  分支差异;代码守卫锁死此行防未来重构退回冻结串丢判定。
- 门禁绿(5 预期 skip);test_hydration 加去物化守卫;版本 2.25→2.26 lockstep。

## 说明

去物化只在 ≥500 行的大 run 触发(小 run 不需要);首批 100 卡不回收以简化
键盘/焦点契约。至此 v2.24(图片内存)+ v2.26(DOM 节点)= 真正窗口化虚拟滚动:
两项资源用量都与 run 大小解耦。
